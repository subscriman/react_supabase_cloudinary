# 프로젝트 복사 후 치환 체크리스트

이 문서는 현재 프로젝트 폴더를 복사해서 새 프로젝트를 만들 때, 그대로 두면 안 되는 항목만 빠르게 점검하기 위한 체크리스트입니다.

## 1. 복사 방식

- [ ] 새 프로젝트 폴더는 현재 프로젝트를 복사하되 `.git`, `node_modules`, `mobile/node_modules`, `mobile/ios/Pods`, `mobile/android/.gradle`, `mobile/.gradle`, `mobile/android/build`, `mobile/android/app/build`, `web/.next`, `.netlify`는 제외
- [ ] 새 프로젝트는 별도 Git 저장소로 시작할지, 기존 저장소를 유지할지 먼저 결정
- [ ] 복사 후 `npm install`, `cd mobile && npm install`, `cd mobile/ios && pod install`을 다시 실행

## 2. 프로젝트 이름 치환

- [ ] 루트 `package.json`의 프로젝트명/설명 점검
- [ ] `mobile/package.json`의 앱 이름 점검
- [ ] `web/package.json`의 웹 프로젝트명 점검
- [ ] `README.md`의 프로젝트명, 저장소 URL, 설명 교체
- [ ] Netlify 사이트명이 바뀌면 `netlify.toml`과 Netlify 연결 상태도 다시 점검

## 3. Android 치환 항목

- [ ] `mobile/android/app/build.gradle`의 `namespace` 교체
- [ ] `mobile/android/app/build.gradle`의 `applicationId` 교체
- [ ] `mobile/android/app/src/main/res/values/strings.xml`의 앱 이름 교체
- [ ] `mobile/android/app/src/main/java/com/subscri/manager/` 패키지 경로를 새 패키지명에 맞게 이동
- [ ] `MainActivity` / `MainApplication` 상단 `package ...` 선언 교체
- [ ] Firebase를 쓰면 `mobile/android/app/google-services.json`을 새 앱 기준으로 다시 발급
- [ ] Google Maps를 쓰면 `GOOGLE_MAPS_API_KEY`를 새 프로젝트 키로 재설정

찾아서 치환할 값:

- [ ] `com.subscri.manager`
- [ ] `subscription-manager-mobile`
- [ ] `subscri`

## 4. iOS 치환 항목

- [ ] Xcode 프로젝트명 `SubscriptionManager`를 새 앱 이름으로 교체
- [ ] `mobile/ios/SubscriptionManager.xcodeproj/project.pbxproj` 내부 타깃명/프로젝트명 점검
- [ ] `mobile/ios/SubscriptionManager/Info.plist`의 `CFBundleDisplayName` 교체
- [ ] iOS Bundle Identifier를 새 값으로 교체
- [ ] Firebase를 쓰면 `mobile/ios/SubscriptionManager/GoogleService-Info.plist`를 새 앱 기준으로 다시 발급
- [ ] 필요 시 `LaunchScreen.storyboard`, 앱 아이콘, 권한 문구 교체

찾아서 치환할 값:

- [ ] `SubscriptionManager`
- [ ] `com.subscri.manager`

## 5. 웹/배포 치환 항목

- [ ] `web/package.json`의 이름과 설명 점검
- [ ] `netlify.toml`의 빌드 설정이 새 프로젝트 구조와 맞는지 확인
- [ ] Netlify 사이트를 새 프로젝트로 다시 연결
- [ ] 기존 Netlify 환경변수가 남아 있으면 새 프로젝트 값으로 교체
- [ ] 저장소를 새로 만들면 GitHub 연동, deploy key, webhook도 다시 연결

찾아서 치환할 값:

- [ ] `subscriman`
- [ ] `react_supabase_cloudinary`
- [ ] 기존 GitHub 저장소 URL

## 6. 환경 변수/시크릿

- [ ] 루트 `.env` 새로 작성
- [ ] `mobile/.env` 새로 작성
- [ ] `web/.env.local` 새로 작성
- [ ] 예시 파일만 유지하고 실제 키 파일은 커밋하지 않도록 확인

새 프로젝트에서 다시 넣어야 하는 값:

- [ ] `NEXT_PUBLIC_SUPABASE_URL`
- [ ] `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- [ ] `SUPABASE_URL`
- [ ] `SUPABASE_ANON_KEY`
- [ ] `NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME`
- [ ] `NEXT_PUBLIC_CLOUDINARY_API_KEY`
- [ ] `NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET`
- [ ] `CLOUDINARY_API_SECRET`
- [ ] `GOOGLE_MAPS_API_KEY`

## 7. 외부 서비스 재연결

- [ ] Supabase 프로젝트를 새로 만들었으면 URL/anon key 교체
- [ ] Cloudinary를 새 프로젝트 기준으로 쓸지 기존 계정을 재사용할지 결정
- [ ] Firebase Android/iOS 앱을 새 패키지명/Bundle ID 기준으로 각각 생성
- [ ] Netlify 사이트를 새 프로젝트로 생성하고 env 등록
- [ ] GitHub 저장소를 새 프로젝트용으로 생성

## 8. 실행 전 최종 확인

- [ ] `nvm use`로 Node 버전 맞춤
- [ ] `cd mobile && npm install`
- [ ] `cd mobile/ios && pod install`
- [ ] `npx react-native run-android`
- [ ] `npx react-native run-ios`
- [ ] `cd web && npm run build`
- [ ] Netlify 또는 배포 대상에서 환경변수 누락 없는지 확인
