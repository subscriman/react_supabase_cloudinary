# ArtTomato 푸시 알림 설계

기준일: 2026-04-03  
범위: 2차 개발 항목 `푸시 알림 설계`

## 1. 목표

- 전시 시작 알림(`exhibition_start_alerts`)을 실제 푸시 채널로 전달할 수 있는 구조를 정의한다.
- 웹 MVP 운영을 깨지 않으면서, React Native(Expo) 앱 확장 시 바로 연결 가능한 설계를 만든다.

## 2. 현재 상태

- 사용자 신청 데이터: `public.exhibition_start_alerts`
- 배치 스크립트: `scripts/alerts/run-start-alerts.js`
- 현재 배치 동작:
  - 발송 대상 계산
  - `sent_at` 마킹
  - 실제 푸시 전송은 아직 미구현

즉, 알림 "대상 선정"은 완료되어 있고, "전송 채널/전송 이력/재시도" 계층이 비어 있다.

## 3. 아키텍처(권장)

1. 대상 선정 배치(`selector`)
- 기존 `run-start-alerts.js`를 확장해 발송 후보를 찾는다.
- 후보를 바로 `sent_at` 처리하지 않고, `notification_deliveries` 큐에 적재한다.

2. 전송 워커(`sender`)
- 새 워커가 `queued` 상태 전달 건을 가져와 채널별(모바일/웹) 전송한다.
- 성공 시 `sent`, 실패 시 `failed` + 재시도 스케줄링을 남긴다.

3. 디바이스/토큰 저장소
- 사용자별 푸시 토큰(`notification_devices`)을 관리한다.
- 무효 토큰은 비활성화해 재시도 비용을 줄인다.

4. 상태 집계
- 특정 `exhibition_start_alerts.id` 기준으로 최소 1건 이상 `sent`이면 `sent_at`를 기록한다.

## 4. 데이터 모델 초안

```sql
-- 사용자 디바이스/푸시 토큰
create table public.notification_devices (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  platform text not null check (platform in ('ios', 'android', 'web')),
  provider text not null check (provider in ('expo', 'fcm', 'webpush')),
  push_token text not null,
  app_version text,
  locale text,
  is_active boolean not null default true,
  last_seen_at timestamptz not null default now(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (provider, push_token)
);

-- 알림 전송 큐/이력
create table public.notification_deliveries (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  start_alert_id uuid references public.exhibition_start_alerts(id) on delete set null,
  exhibition_id uuid references public.exhibitions(id) on delete set null,
  channel text not null check (channel in ('mobile_push', 'web_push')),
  provider text not null check (provider in ('expo', 'fcm', 'webpush')),
  template_key text not null,
  payload jsonb not null,
  status text not null default 'queued'
    check (status in ('queued', 'processing', 'sent', 'failed', 'cancelled')),
  retry_count int not null default 0,
  next_retry_at timestamptz,
  provider_message_id text,
  error_code text,
  error_message text,
  sent_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

RLS 원칙:

- `notification_devices`: 본인 조회/등록/비활성화 허용
- `notification_deliveries`: 본인 조회(읽기 전용), 쓰기는 관리자/서버 워커만 허용

## 5. 환경 변수 초안

- `EXPO_ACCESS_TOKEN` (Expo Push API 사용 시)
- `FCM_SERVICE_ACCOUNT_JSON` (FCM 직접 송신 시)
- `WEB_PUSH_PUBLIC_KEY`
- `WEB_PUSH_PRIVATE_KEY`
- `WEB_PUSH_SUBJECT`

1차 구현에서는 `EXPO_ACCESS_TOKEN`만 필수로 시작하고, 웹 푸시는 2단계로 분리한다.

## 6. 발송/재시도 정책

- 기본 재시도 횟수: 최대 5회
- 백오프: 1분 → 5분 → 30분 → 6시간 → 24시간
- 영구 실패 코드(토큰 만료/해지 등)는 즉시 `notification_devices.is_active=false`
- 전시 시작일이 지난 건은 `cancelled` 처리

## 7. 이벤트/모니터링

- 분석 이벤트:
  - `push_opt_in`
  - `push_delivery_queued`
  - `push_delivery_sent`
  - `push_delivery_failed`
- 운영 지표:
  - 큐 적체 수(`queued`)
  - 성공률(`sent / total`)
  - 토큰 무효율(만료/해지 비율)

## 8. 단계별 구현 순서

1. DB 마이그레이션
- `notification_devices`, `notification_deliveries` + 인덱스 + RLS

2. API
- 디바이스 등록/해제 API (`/api/notifications/devices`)

3. 배치
- selector: 기존 `run-start-alerts.js`를 큐 적재 방식으로 개편
- sender: `scripts/alerts/run-push-deliveries.js` 추가

4. 모바일 앱 연동
- Expo Notifications 토큰 등록
- 앱 상태별 처리(foreground/background/open)

5. 웹 푸시 확장(선택)
- 서비스워커 + webpush 구독/해지

## 9. QA 기준

- 같은 유저/같은 전시 알림이 중복 발송되지 않는다.
- 무효 토큰이 자동 비활성화된다.
- 워커 재실행 시 idempotent하게 처리된다.
- `sent_at`는 실제 발송 성공 이후에만 채워진다.
